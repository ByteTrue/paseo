## Background

The first front-end custom-provider attempt exposed the raw provider override model directly to users: base provider, provider id, arbitrary environment variables, and model rows. That made the UI look like a generic `agents.providers` editor instead of the workflow users actually wanted: configure a Claude Code-compatible endpoint once, then pick it like a named Claude entry when launching agents.

Claude Code-compatible third-party endpoints such as DeepSeek, Z.AI, Qwen, and proxies are normally selected through environment variables. For the first app-managed version, the requirement was to support that common Claude path without turning the UI into a general custom-provider editor.

## Decision

Use **Claude endpoint variants** as the user-facing abstraction for first-version endpoint customization:

- The UI is Claude-only and lives under **Settings → Providers → Claude → Claude endpoints**.
- Each variant persists as an `agents.providers.{id}` override with `extends: "claude"`.
- UI-managed variants carry `params.paseoManagedKind: "claudeEndpointVariant"`.
- The form exposes exactly these six optional environment fields:
  - `ANTHROPIC_BASE_URL`
  - `ANTHROPIC_AUTH_TOKEN`
  - `ANTHROPIC_DEFAULT_OPUS_MODEL`
  - `ANTHROPIC_DEFAULT_SONNET_MODEL`
  - `ANTHROPIC_DEFAULT_HAIKU_MODEL`
  - `CLAUDE_CODE_SUBAGENT_MODEL`
- Blank env fields are omitted; `ANTHROPIC_MODEL` is not generated by the UI-managed flow.
- Saved variants appear as Claude-like provider options when creating agents, while preserving their own provider id for launch.
- The generic front-end "Add custom provider" UI stays removed. Raw provider overrides remain an advanced config-file capability, not the ordinary app UI.

## Rationale

This shape matches the user mental model: "add another Claude endpoint" rather than "create an arbitrary provider override". It also reuses the existing provider override and derived-provider runtime path, so daemon launch behavior remains simple: a selected variant is still backed by the Claude adapter with its configured env overlay.

The managed marker prevents the app from accidentally listing or rewriting hand-written advanced `extends: "claude"` providers that may contain custom commands, params, or unusual env. Optional provider snapshot metadata (`derivedFromProviderId`, `managedKind`) lets the app render a variant with Claude visual treatment without changing the launch provider id.

## Alternatives Considered

- **Generic custom-provider form in the app**: rejected because it exposes internal config concepts (`extends`, base provider, provider id, raw env) and caused the original UX mismatch.
- **Model-level endpoint rows under Claude models**: rejected because Claude endpoint selection is launch-environment scoped, not just a model id choice.
- **Main/Fast model simplification**: rejected because the user wanted the explicit Claude Code env fields exposed one-for-one, except `ANTHROPIC_MODEL`.
- **Save-time endpoint testing**: rejected because built-in providers do not have a settings-time test flow, and adding one would over-design this feature.

## Consequences

- Future endpoint work for Codex/OpenCode/Pi should not be smuggled into this Claude v1 UI. It should get its own design and its own provider-specific settings surface.
- Hand-written provider overrides stay supported, but the app only manages entries marked with `params.paseoManagedKind: "claudeEndpointVariant"`.
- Deleting a variant removes the provider override; agents or preferences pointing at that provider id may need another provider selected before reuse or resume.
- Provider snapshot entries may carry optional derived-provider metadata for app rendering. These fields are compatibility-safe and do not change provider probing semantics.
