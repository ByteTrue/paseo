# Daemon clients require explicit authorization

Paseo will treat the daemon administrator password as the secret that authorizes new remote clients, not as a value embedded in a pairing link. A pairing link is a long-lived bootstrap URL for discovery and daemon identity, not a complete authorization grant; remote clients prove ongoing authorization with an independent Ed25519 client keypair that the daemon has enrolled. Each client stores a separate auth keypair per daemon/host so revocation, compromise, and cross-daemon correlation stay scoped to one `serverId`. Localhost and IPC connections remain trusted for ordinary daemon operator traffic because local shell/filesystem access is already equivalent to daemon-user authority, but auth administration is a separate class of action and requires the administrator password once one exists.

## Considered Options

- Keep the current pairing-link-as-authorization model. Rejected because anyone with the link keeps full control and the daemon password remains marginal.
- Auto-migrate old remote clients. Rejected because it would preserve old pairing links as complete grants and weaken the redesign.
- Use bearer tokens or reuse relay E2EE keys for client identity. Rejected because authorization should be a distinct client signing identity that can be proven and revoked independently.

## Consequences

Relay and non-loopback direct-TCP clients must enroll with the daemon administrator password before receiving normal daemon data or issuing control RPCs. Pre-authorization connections expose only minimal daemon identity and an auth challenge; agents, workspaces, providers, and status snapshots are sent only after authorization succeeds. Enrolled clients prove authorization on every connection by signing a fresh daemon challenge with their per-daemon/host Ed25519 auth keypair, so browser reconnects remain automatic and do not prompt for the administrator password. Legacy direct-TCP password credentials are removed: clients stop writing or using host `password`, `tcp://...?password=...`, and client-side `PASEO_PASSWORD` as connection grants, though parsers may accept and discard legacy fields for compatibility. An authorized client has full ordinary daemon operator authority in the first version of this system; Paseo is not introducing role-based access control for normal daemon operations as part of this redesign. If no daemon administrator password is configured, remote enrollment is disabled and remote clients can only see guidance to set a password locally; a pairing link must never let a remote client set the first administrator password. Authorized clients may view or copy the long-lived pairing link without entering the administrator password because the link is bootstrap material, not an authorization grant. Authorized clients are persisted in a separate private `$PASEO_HOME/authorized-clients.json` file instead of `config.json`, because client authorization is runtime state rather than daemon configuration. Authorized-client management surfaces may be shown to any authorized client, local or remote, but auth administration mutations such as revoking clients or changing the administrator password require the administrator password every time. Administrator-password submission is allowed only over relay E2EE, IPC/localhost, or TLS direct TCP; non-loopback cleartext direct TCP must reject enrollment and auth administration. Remote administrator-password failures are rate-limited by remote source and client public key with exponential backoff; localhost/IPC remains lenient and primarily logs failures. Changing the daemon administrator password automatically revokes all enrolled remote clients, forcing them to re-enroll with the new password. Revoking an authorized client immediately closes its active remote sessions; changing the administrator password immediately closes all active remote authorized sessions. Environment variables, TTY state, command origin, and localhost appearance are not security boundaries; users should only enter the administrator password on devices they trust.
